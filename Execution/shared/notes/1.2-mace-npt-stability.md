## Approach

The Sub 1.1 hybrid NVT script (`phases/phase-1/subphase-1.1/output/scripts/mace_hybrid_nvt.py`)
was forked to `phases/phase-1/subphase-1.2/output/scripts/mace_hybrid_npt.py`
with these additions:

1. **`MonteCarloBarostat(1 atm, 300 K, freq=25 steps)`** added to the hybrid
   system. Toggled disabled (`freq=0`) during NVT stage and re-enabled during
   NPT stage via `setFrequency()`. Standard OpenMM NPT pattern.
2. **Two-stage equilibration:** 50 ps NVT at dt=0.5 fs (safer warmup for hybrid
   systems; avoids the Sub 1.1 `maxIterations=200` NaN regression) + 200 ps NPT
   at dt=1 fs (full volume relaxation before production).
3. **OpenMM checkpoint/restart:** `Simulation.saveCheckpoint()` /
   `.loadCheckpoint()` via `CheckpointReporter(path, 100000)` (~100 ps wall).
   On each SLURM session, the script detects `checkpoint_latest.chk` and
   resumes production; otherwise starts fresh.
4. **Append-mode reporters:** `DCDReporter(..., append=True)` + file-handle
   `StateDataReporter` with suppressed duplicate header. Multiple SLURM
   resubmissions concatenate to one DCD + one CSV per protein.
5. **Walltime guard at 22.5 hr (81,000 s)** so the script always has time to
   save a clean final checkpoint and update `progress.json` before SLURM
   hard-kill at 23:59:00.
6. **All Sub 1.1 safeguards retained verbatim:** `PYTHONNOUSERSITE=1`,
   `maxIterations=2000` minimization + post-min max-force <1e5 check, GPU
   keepalive thread (5-min matmul cadence), OpenCL platform only (CUDA
   broken on all available GPUs per Sub 1.1 Subagent A).

## §3b CUDA optimization patch resubmit (2026-04-20)

All prior MACE H200 production attempts (v2-v9, job IDs 8893817-8939397) were
YCRC auto-cancelled at 2-3 hours due to 0% GPU utilization. Root cause:
`matscipy.neighbour_list` holds the Python GIL during its C extension call
(~50% of per-step wall time), idling the GPU. The in-Python keepalive thread
cannot execute during GIL-blocked periods. External-process keepalive was
deployed but also proved insufficient for YCRC's sampling window.

**Fix deployed:** `mace_cuda_patch.py` replaces the openmmml `_computeMACE`
callback with a CUDA-native version:
1. GPU neighbor list via `torch.cdist` (replaces matscipy)
2. MACE model converted to cuequivariance format (accelerated tensor products)
3. All tensors stay on GPU (no CPU→GPU round-trips per step)
4. Safety fallback to matscipy if any atom drifts within cutoff of box edge

Packages added to env-mace: cuequivariance 0.9.1, cuequivariance-torch 0.9.1,
torch-cluster 1.6.3+cu121. Self-diagnostic built into production script
(10 steps post-minimization, NaN check before committing to full production).

## §5 NPT Investigation Results (2026-04-28)

### Summary

**MACE hybrid NPT with MonteCarloBarostat is fundamentally unstable.**
Float64 MACE delays the crash but does not prevent it. The root cause is
the interaction between MC barostat box rescaling and the MACE hybrid force
computation, not solely numerical precision.

### Diagnostic Results

| Test | Config | Crash point | T (K) | Density (g/cm³) | Verdict |
|------|--------|------------|-------|-----------------|---------|
| A (q6byygle) | Classical-only NPT, freq=25 | Never (80 ps clean) | 292-307 | 1.019-1.035 | **PASS** |
| B (q6wjsyhu) | MACE hybrid f32, freq=100 | ~5 ps equil | — | — | **FAIL** |
| F (q6q6ijfi) | MACE hybrid f64, freq=25 | ~17 ps equil | 299-311 | 1.020-1.033 | **FAIL** |
| Original (d6v6fxwq) | MACE hybrid f32, freq=25 | ~25 ps equil | — | — | **FAIL** |

### Key Findings

1. **Classical NPT is rock-solid** (Test A PASS). The system build, solvation,
   ions, box size, and MC barostat all work correctly without MACE. The problem
   is exclusively in the MACE hybrid potential interaction with the barostat.

2. **Barostat frequency does not help** (Test B). Reducing MC move frequency
   from 25 to 100 (4× less frequent) did not prevent the crash — it actually
   crashed earlier (~5 ps vs ~25 ps). This rules out "too-frequent MC moves"
   as the root cause.

3. **Float64 helps but does not fix** (Test F). f64 MACE survived 15 ps with
   3 healthy checkpoints (T=299→311→302 K, density stable ~1.03 g/cm³) before
   crashing at ~17 ps. f32 Test B crashed within 5 ps. This confirms precision
   is a contributing factor but not the sole cause.

4. **The crash is stochastic but inevitable.** Different configurations crash
   at different times (5 ps, 17 ps, 25 ps), suggesting the NaN is triggered
   when an MC barostat move happens to create a geometry where the MACE model
   produces discontinuous or unphysical forces. This accumulates over many
   barostat moves until a catastrophic failure.

5. **Root cause identified: missing minimum-image wrapping in `_computeMACE`.**
   The openmm-ml `_computeMACE` callback (macepotential.py:232-267) does NOT
   apply minimum-image wrapping to protein atom positions before building the
   neighbor list. During MC barostat trial moves, OpenMM may wrap positions
   differently than expected. If the protein centroid drifts near a box face,
   some atoms can end up on the opposite side of the periodic box after
   wrapping. The neighbor list builder then produces an incorrect edge graph
   (bonded atoms appear box_length apart instead of 1.5 Å), and MACE
   computes pathological forces → NaN. This is related to openmm-ml issue #52
   (PBC not replicated for ML portion). A ~3 line fix (applying fractional
   coordinate wrapping before `get_neighborhood`) may resolve this — untested.
   See `output/scripts/npt_diagnostics/npt_investigation_results.md` §2 for
   full source-code analysis and proposed patch.

6. **No published stable MACE NPT via OpenMM MC barostat.** Literature survey
   found that all stable MACE NPT simulations used continuous barostats
   (Nose-Hoover + Parrinello-Rahman) in non-OpenMM engines (SchNetPack, ASE).
   OpenMM has no continuous barostat option. The NNP/MM paper (Galvelis 2023,
   JCTC) deliberately chose NVT for all hybrid ML/MM production runs.

## §7 RESOLUTION 2026-05-02: MACE NPT FIXED via Round 3 recipe

This note's earlier "NPT not viable" verdict was based on tests that did not
address the actual root cause. After a literature SubAgent identified
**openmm-ml issue #91** (jchodera, 2025-02-04, still open) — the
`_removeBonds(allInSet=True)` call inside `createMixedSystem` strips all
classical bonds for the ML region, and OpenMM's `findMolecules()` then sees
each protein atom as a singleton "molecule," causing `MonteCarloBarostat` to
scale each protein atom independently around the box origin — a Round 3
iteration discovered that a second issue compounds the failure: the same
bond stripping leaves protein C-H/N-H/O-H bonds **unconstrained**, and at
dt=1 fs the C-H stretching mode is at the edge of velocity-Verlet stability.

**The production recipe is:**

1. `MLPotential('mace-off24-medium').createMixedSystem(...)` (standard)
2. `add_protein_sentinel_bonds(hybrid_system, topology, protein_atoms)` —
   adds zero-strength `HarmonicBondForce` along the protein bonded graph.
   Fixes the singleton-molecule bug.
3. `add_protein_hbonds_constraints(hybrid_system, topology, protein_atoms)` —
   re-adds AMBER14 H-bond rigid constraints for the protein region. Fixes
   the unconstrained-H instability at dt=1 fs.
4. `MonteCarloBarostat(1 atm, 300 K, freq=25)` (standard)
5. `LangevinMiddleIntegrator(300 K, 1 ps⁻¹, dt=1 fs)` (standard)

Both helpers are in `output/scripts/npt_diagnostics/npt_diag_common.py`.
Reproducible from `test_L_hbonds.py` (25 ps gpu_devel) and `test_P_extended.py`
(100 ps priority queue confirmation, job 10441986).

**Empirical evidence:**
- test_J (Round 2, sentinel only): NVT 10 ps PASS, NPT 12.5 ps NaN
- test_K (Round 3A, sentinel + dt=0.5): NVT 1 ps + NPT 13 ps PASS (full target)
- test_L (Round 3B, sentinel + HBonds + dt=1): NVT 5 ps + NPT 25 ps PASS
- test_M (Round 3C, sentinel + HBonds + dt=0.5): NVT 1 ps + NPT 14 ps PASS
- **test_P (Round 3D, sentinel + HBonds + dt=1, priority queue): NVT 1 ps + NPT 100 ps PASS**
  Final density 1.041 g/cm³, T 297.6 K, PE stable around -364,500 kJ/mol.

**Sub 1.4 production:** MACE NPT (NOT NVT). Recipe per above. Throughput
~0.91 ns/day on RTX 5000 Ada, ~2.56 ns/day on H200 OpenCL hybrid.

The two-stage NVT fallback (`output/npt_nvt_production_plan.md`) is
