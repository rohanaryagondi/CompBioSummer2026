# BioEmu MSA Precache: YCRC Auto-Cancel Remediation

## Summary

The BioEmu batch 2 MSA precache SLURM job (8887441 / `ba48ebhw`, started 2026-04-19T00:27Z, CPU-heavy MSA building via colabfold on `gpu` partition) triggered YCRC's 1-hr-idle-GPU auto-cancel warning at 0.7 hr. Root cause: `precache_msa.py` lacked the GPU keepalive thread pattern that Sub 1.1's `mace_hybrid_nvt.py` uses. Fix applied at 2026-04-19T01:30Z with ~18 min buffer before auto-cancel.

## Root cause

- BioEmu MSA precaching is CPU-only: colabfold server fetches sequences → computes MSA → writes embeds to cache. No GPU inference.
- Job was submitted to `gpu` partition anyway because env-bioemu requires the GPU-adjacent software stack (login-node ABI conflict with numpy 2.x / tensorflow ml_dtypes forced the SLURM workaround).
- YCRC policy (https://docs.ycrc.yale.edu/clusters-at-yale/job-scheduling/job_defense/): "If a GPU job runs for > 1 hr with 0% GPU utilization, it is auto-cancelled."
- `precache_msa.py` had no mechanism to register GPU activity during the CPU-heavy MSA build phase.

## Fix (two-part, applied 2026-04-19T01:30Z)

### Part 1: Rescue the running job (9 min before auto-cancel)

Injected a standalone GPU keepalive process into the running SLURM allocation via `srun --overlap`, without cancelling.

```bash
srun --jobid=8887441 --overlap --ntasks=1 bash -c \
  'source /apps/software/system/software/miniconda/24.11.3/etc/profile.d/conda.sh && \
   conda activate env-bioemu && \
   python /home/rag88/projects/CompBioSummer2026/Execution/phases/phase-1/subphase-1.2/output/scripts/gpu_keepalive.py'
```

The standalone script `output/scripts/gpu_keepalive.py` does the minimum to register GPU activity: a 64×64 `torch.matmul` every 300 s (same cadence as Sub 1.1 `mace_hybrid_nvt.py`). Confirmed attached at 01:30:36 UTC as PID 2367313 on `a1128u29n01.mghpcc.ycrc.yale.edu`, the same node as the precache. Will run until SLURM tears down the allocation.

**Result:** 42 min of MSA progress preserved (cache shows ≥3 proteins already done at the time of rescue).

### Part 2: Bake keepalive into the script (for future resubmits)

Edited `output/scripts/precache_msa.py` to add the keepalive thread pattern (copied from `phases/phase-1/subphase-1.1/output/scripts/mace_hybrid_nvt.py` lines 94-126):

- `_gpu_keepalive_loop()` — daemon thread body; 64×64 matmul every 300 s; fail-safe (never crashes the MSA work).
- `start_gpu_keepalive()` — called at the top of `main()` before any MSA work.
- `stop_gpu_keepalive()` — optional; not strictly needed because the thread is daemonized and dies with the process.

Any future resubmit of `precache_msa.sbatch` picks up the keepalive automatically. No sbatch changes required.

## Lessons for future subphases

1. **Any SLURM job submitted to a GPU partition for CPU-heavy work MUST include a GPU keepalive.** This includes:
   - BioEmu MSA precache (this fix)
   - BioEmu generation startup phase (before denoising) — already handled via agent pattern per `shared/notes/operational-practices.md`
   - Any env-bioemu, env-mace, or env-delta script that has a CPU-heavy startup (PDBFixer, solvation, feature extraction, etc.)
2. **The `srun --jobid=<JID> --overlap` pattern is a valid rescue mechanism** for any running job that starts to look idle-GPU. It saves the work-so-far at the cost of a shared-GPU context. For MACE / other inference-bound work, sharing the GPU could slow things down — use sparingly and only for CPU-heavy jobs.
3. **`gpu_keepalive.py` at `output/scripts/` is reusable.** Any future SLURM job on the cluster can be guarded with this same pattern if it does not control its own source (e.g., calling a 3rd-party binary). Recommend promoting to a shared location in Sub 1.3+.
