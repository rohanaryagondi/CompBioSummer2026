# MACE-OFF24 CUDA Benchmark (Option 2 — Completed)

## TL;DR

- **Interop fix: PARTIALLY WORKING, mode-dependent, not robust.** Fix A+B
  (device pin + monkey-patch `torch.cuda.synchronize()` around
  `_computeMACE`) unblocks the simplest probe case (short minimize + a few
  hundred steps on pure-MACE vacuum with `createMixedSystem` path on
  RTX 5000 Ada). It FAILS under any more realistic regime: long
  minimization (2000-iter), hybrid systems with TIP3P water, or on H200
  instead of RTX.
- **Fix applied:** (A) `CUDA_VISIBLE_DEVICES=0` + `torch.cuda.set_device(0)` +
  `device='cuda:0'` passed to `MLPotential.createSystem/createMixedSystem`,
  (B) runtime monkey-patch of `openmmml.models.macepotential._computeMACE`
  to call `torch.cuda.synchronize()` before and after the MACE forward
  pass. Fix A alone is insufficient (probe's post-minimize getState still
  crashes, SLURM 8837310).
- **Crambin vacuum RTX 5000 Ada CUDA: 0.142 ns/day (measured) vs OpenCL
  0.142 ns/day → 1.0× (NO speedup).** MACE-OFF24 on RTX 5000 Ada is
  inference-bound: MACE forward pass dominates per-step cost; OpenMM
  integrator kernel speed is irrelevant.
- **Hybrid WW RTX 5000 Ada CUDA: FAILED** (post-min getState crash,
  SLURM 8838683).
- **Crambin vacuum H200 CUDA: FAILED** (post-min getState crash,
  SLURM 8839602 with createSystem, 8839764 with createMixedSystem-all-atoms).
- **Concurrent H200 OpenCL hybrid WW (SLURM 8789805): 2.12 ns/day stable
  over 7 ps equilibration before wall cancellation** — confirms Subagent J's
  extrapolation. This is the real Phase 2 throughput number for Option 5.
- **Phase 2 recommendation: REJECT Option 2 as Phase 2 primary. Proceed
  with Option 5 (H200 OpenCL) as the Phase 2 MLFF workhorse.** Rationale
  below.

## Interop diagnosis

### Reproduction

Subagent L's `cuda_interop_probe.py` reproduces Subagent J's crash on
RTX 5000 Ada:

```
Platform CUDA ACTIVE: probe step in ~1000 ms, PE=-272269 kJ/mol
...
minimizeEnergy(2000 iter) → RuntimeError in TorchScript:
    "CUDA driver error: invalid resource handle"
```

at the first non-trivial `minimizeEnergy()` call on a system with MACE
wired in via `openmmml.MLPotential` → `openmm.PythonForce`.

### Root cause

openmm-ml 1.6 implements MACE forces through `openmm.PythonForce`, which
invokes a Python callback from OpenMM's C++ integrator on each force
evaluation. Inside the callback, MACE runs a TorchScript-compiled forward
pass on PyTorch's CUDA context (per-thread default stream, PyTorch caching
allocator). OpenMM's CUDA plugin uses a different stream. There is no
built-in synchronization between them. When OpenMM reads force buffers
written asynchronously by PyTorch — and in the minimize / getState / post-
minimize getState path, this reading happens before PyTorch's work has
actually completed — the result is the `invalid resource handle` error.

This is the canonical "dual CUDA context" failure mode for
PythonForce-based MLFF wrappers. The clean fix is `openmm-torch`'s
`TorchForce` (native C++ CUDA integration that shares OpenMM's stream).

Attempts to use conda-forge `openmm-torch` 1.5.1 fail ABI compatibility:
conda-forge builds pin `openmm >=8.4.0,<8.5.0a0`, incompatible with our
source-built OpenMM 8.5.1 (Subagent J). A from-source openmm-torch build
would take 1-2 hrs and is NOT undertaken in this session per escalation
protocol.

## Fix applied

### Fix A — device alignment (necessary, not sufficient)

Before the first torch import:

```python
os.environ['CUDA_VISIBLE_DEVICES'] = '0'
```

After importing torch:

```python
torch.cuda.init()
torch.cuda.set_device(0)
```

Pass `device='cuda:0'` to the MLPotential call:

```python
potential.createMixedSystem(topology, mm_system, atoms,
                            interpolate=False, device='cuda:0')
```

OpenMM CUDA platform properties:

```python
{"Precision": "mixed", "DeviceIndex": "0", "DeterministicForces": "false"}
```

Empirical: Fix A alone still crashes at post-minimize getState
(SLURM 8837310 after 40 s).

### Fix B — runtime sync around MACE (necessary, not sufficient alone)

Monkey-patch `openmmml.models.macepotential._computeMACE` to sync CUDA
before and after the forward pass:

```python
import openmmml.models.macepotential as mp
_orig = mp._computeMACE
def _computeMACE_sync(state, **kw):
    import torch
    if torch.cuda.is_available():
        torch.cuda.synchronize()
    energy, forces = _orig(state, **kw)
    if torch.cuda.is_available():
        torch.cuda.synchronize()
    return energy, forces
mp._computeMACE = _computeMACE_sync
```

### Where A+B works and where it fails (empirical matrix)

| Case | Min iter | Platform | System built via | Steps after min | Result |
|------|----------|----------|------------------|-----------------|--------|
| Probe 1 (SLURM 8837309) | 10 | RTX CUDA | createMixedSystem, all-atoms | 10 | PASS (22 s wall) |
| Probe 2 (SLURM 8838060) | 10 | RTX CUDA | createMixedSystem, all-atoms | 10+100+500=610 | PASS (cancelled before 1000-step chunk) |
| Probe 3 vacuum bench (SLURM 8837567) | 2000 | RTX CUDA | createSystem, pure MACE | ~500 (before 1st reporter) | FAIL: illegal memory access in MACE cat() |
| Probe 4 hybrid bench (SLURM 8838683) | 2000 | RTX CUDA | createMixedSystem, 534 MACE / 6962 TIP3P | 0 | FAIL: invalid handle at post-min getState |
| Probe 5 vacuum H200 (SLURM 8839602) | 2000 | H200 CUDA | createSystem, pure MACE | 0 | FAIL: invalid handle at post-min getState |
| Probe 6 vacuum H200 v2 (SLURM 8839764) | 2000 | H200 CUDA | createMixedSystem, all-atoms | 0 | FAIL: invalid handle at post-min getState |

**Pattern:** A+B is a fragile fix that only works when (1) minimization is
short (≤10 iter), (2) the system is pure MACE with no classical PME water,
(3) the GPU is RTX 5000 Ada. Any deviation (longer minimize, hybrid
solvent, different GPU) reopens the crash.

The fix is not viable for production Phase 2 use, which requires deep
minimization on hybrid systems on H200.

## Throughput benchmarks

### RTX 5000 Ada

| System | Platform | ns/day | Speedup vs RTX OpenCL | Source |
|--------|----------|--------|------------------------|--------|
| Vacuum crambin (642 atoms) | OpenCL | 0.142 | 1.0× | Subagent J SLURM 8761627 |
| Vacuum crambin (642 atoms) | CUDA (Fix A+B, probe) | **0.142** | **1.0× (NO speedup)** | Subagent L SLURM 8838060, 610 steps / 370 s |
| Hybrid WW (534/7,496 atoms) | OpenCL | 0.184 | 1.0× | Subagent G SLURM 8736919 |
| Hybrid WW (534/7,496 atoms) | CUDA | FAIL | — | Subagent L SLURM 8838683 |

**Key finding:** CUDA gives NO speedup over OpenCL for MACE-OFF24 on
RTX 5000 Ada vacuum crambin. The MACE forward pass is the bottleneck;
OpenMM integrator kernel overhead is negligible. Switching OpenMM's
integrator from OpenCL to CUDA changes nothing for the MLFF-bound
workload. This corroborates Subagent K's Option 4 finding (implicit
solvent gave only 1.07× speedup → MACE inference is the dominant cost).

### H200

| System | Platform | ns/day | Note | Source |
|--------|----------|--------|------|--------|
| Vacuum crambin (642 atoms) | OpenCL | 1.552 | ~11× RTX OpenCL | Subagent J SLURM 8761949 |
| Vacuum crambin (642 atoms) | CUDA | FAIL | invalid handle at post-min getState | Subagent L SLURM 8839602, 8839764 |
| Hybrid WW (534/7,598 atoms) | OpenCL | **2.12** | confirms SubJ extrapolation | SLURM 8789805 (7 ps equil, cancelled at 2.5 h wall) |
| Hybrid WW (534/7,598 atoms) | CUDA | NOT ATTEMPTED | vacuum CUDA already failed on H200 | — |

H200 OpenCL hybrid WW achieved 2.11-2.13 ns/day stably through 7 ps of
equilibration. **This closes the Option 5 validation: Phase 2 MACE at
IP scope (9 proteins × 2 MLFFs × 10 ns) is FEASIBLE on H200 OpenCL at
~3,300 GPU-hrs total, well within the revised 47,300 GPU-hr Phase 2
MLFF budget.**

### What the H200 CUDA data would have shown (had it worked)

Given the RTX 5000 Ada CUDA=OpenCL parity for MACE on vacuum crambin, we
expect H200 CUDA would also ≈ H200 OpenCL (~1.5 ns/day vacuum), not
substantially faster. The MACE inference bottleneck is hardware-agnostic
in this sense: it shows up on every GPU we've tested.

If, hypothetically, a future openmm-torch rebuild enables CUDA on H200,
the expected speedup over H200 OpenCL is 1-2× at most — well below the
≥5× needed for ACCEPT.
