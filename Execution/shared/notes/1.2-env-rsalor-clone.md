# Cross-Agent Note: env-bioemu-rsalor venv split + ProteinGym MSA precache (Sub 1.3 Gamma unblock)

## Finding

The Sub 1.2 RSALOR feature pipeline shipped a script (`compute_rsalor_batch1.py`)
with two regimes: RSA-only (default, no MSAs) and FULL (RSA + LOR via the `rsalor`
Python package). Sub 1.2 closed in RSA-only mode because (i) ProteinGym MSAs were
not on the filesystem and (ii) `rsalor` was not installed in any conda env.

This single-shot rescue task (2026-05-06) cleared both blockers and populated
the LOR/RSALOR/n_msa_sequences columns for **46 of 47** batch 1 proteins (100%
of currently active batch 1; the only excluded protein is YAP1_HUMAN which was
dropped during BioEmu generation due to a 0.7% physicality pass rate).

### What was done

1. **MSA precache.** Downloaded the ProteinGym v1.3 MSA bundle
   (`DMS_msa_files.zip`, 1.5 GB compressed / 5.2 GB unzipped) from
   `https://marks.hms.harvard.edu/proteingym/ProteinGym_v1.3/DMS_msa_files.zip`
   to scratch:

   ```
   /nfs/roberts/scratch/pi_mg269/rag88/proteingym/msas/DMS_msa_files/    (194 .a2m files)
   /nfs/roberts/scratch/pi_mg269/rag88/proteingym/msas/by_uniprot/       (47 symlinks named <UNI>.fasta)
   ```

   Each batch 1 UniProt ID has a symlink in `by_uniprot/<UNI>.fasta` pointing at
   the matching `.a2m` file in `DMS_msa_files/` (matched by `<UNI>_*.a2m` prefix,
   2023-08-07 version preferred over 2021 when both exist for DYR_ECOLI and
   HXK4_HUMAN; SPG1_STRSG has only the 2021 version). The `.a2m` format is
   FASTA-syntactic (lowercase = insert states); `rsalor` reads it transparently.

2. **Environment split via venv (NOT conda clone).** Conda clone of `env-bioemu`
   was OOM-killed twice on the login node — the conda fork ballooned during the
   12 GB env replication. Per the operational-practices fallback, used a Python
   venv with `--system-site-packages`:

   ```bash
   /home/rag88/.conda/envs/env-bioemu/bin/python -m venv \
       --system-site-packages /home/rag88/.conda/envs-venv/env-bioemu-rsalor
   ```

   This creates a thin venv whose Python sees env-bioemu's full site-packages
   (torch 2.7.1+cu128, bioemu, biopython 1.87, etc.) plus anything `pip install`-ed
   into the venv. `rsalor 1.1.9` was installed via the venv's pip and resides
   under `~/.local/lib/python3.10/site-packages/rsalor/` (user-site; the venv's
   pip falls through to user-site by default on this system).

   **Activation pattern (NOT `conda activate`):**
   ```bash
   PYTHONNOUSERSITE=0 /home/rag88/.conda/envs-venv/env-bioemu-rsalor/bin/python <script.py>
   ```
   The `PYTHONNOUSERSITE=0` is required because `rsalor` lives in user-site;
   setting it to 1 (as done in some templates) hides rsalor.

3. **Sibling script (`compute_rsalor_full_batch1.py`).** The original
   `compute_rsalor_batch1.py` is preserved untouched (per task constraint: "DO
   NOT modify the existing compute_rsalor_batch1.py script"). The original's
   anticipated rsalor API was wrong (`get_scores()` rows use `mutation_fasta`
   keys like `"K1A"`, not separate `position`/`wt_aa`/`mut_aa` fields, and the
   score columns are `LOR` / `RSA*LOR` capitalized; MSA depth is `m.depth`).
   A sibling at the same scripts path implements the correct parsing and writes
   the same output schema. Output paths (per-protein `<UNI>_rsalor.{csv,npz}`
   and aggregated `batch1_rsalor.csv` / `batch1_rsalor_summary.csv`) are
   unchanged from the original.

4. **Memory-isolation execution pattern.** The login node's OOM-killer
   terminated a single-process pass through all 47 proteins around protein 28
   because some MSAs are very large (PHOT_CHLRE: 1.6M sequences / 245 MB;
   ENVZ_ECOLI: 1.9M sequences / 173 MB), and rsalor's C++ extension does not
   release memory aggressively between MSAs in one process. Worked around by
   invoking the script once per protein with `--only <UNI>` (47 subprocess
   invocations, each freed at process exit). Per-protein wall time: 0.2–52 s.
