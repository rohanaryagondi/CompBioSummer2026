# Cross-Agent Note: MACE H200 Throughput Ceiling Analysis

## Finding

Extensive benchmarking on H200 (2026-04-22 through 2026-04-24) confirms **2.56
ns/day is the throughput ceiling** for MACE-OFF24-medium hybrid NPT on H200
OpenCL with the current software stack (torch 2.10, f32 bypass, vesin NL).
This is a **1.25x improvement** over the original 2.04 ns/day baseline. Eight
optimization paths were tested; two succeeded (f32 bypass, vesin NL), six
hit dead ends.

## Bottleneck Breakdown (per-step, WW domain 534 protein atoms)

| Component | Time (ms) | % of step |
|-----------|-----------|-----------|
| MACE e3nn float64 inference | 25-30 | 58-70% |
| PythonForce callback overhead | 5-10 | 12-23% |
| matscipy neighbor list (CPU) | 1-2 | 2-5% |
| OpenMM OpenCL integrator | 1-2 | 2-5% |
| **Total** | **43-44** | **100%** |

The dominant bottleneck is MACE neural network inference through e3nn tensor
products (~40ms of ~43ms step). Even with f32 (which cut inference from ~30ms
to ~25ms), this remains >90% of step cost. Classical OpenMM forces are <3ms.

## Optimizations Tested

### Successful (applied to production)

| Path | Result | Speedup |
|------|--------|---------|
| Float32 bypass (custom PythonForce) | Bypasses openmmml, loads model ourselves as f32 | 2.56 vs 2.04 ns/day = **1.25x** |
| Vesin NL (replacing matscipy) | Eliminates GIL deadlock, prevents YCRC auto-cancel | Reliability fix (no ns/day change) |

### Dead Ends (investigated, not feasible)

| Path | Result | Blocker |
|------|--------|---------|
| GPU neighbor list (torch minimum-image) | Forces match to 1e-15 eV/Å; 2.5% speedup | matscipy is only 1-2 ms/step on H200 |
| OpenMM CUDA platform | CUDA_ERROR_UNSUPPORTED_PTX_VERSION | cuda-nvrtc 13.2 PTX too new for CUDA 12.8 driver; symlink trick fails on SONAME version check |
| openequivariance (oeq) conversion | API mismatch: MACE passes 4 args, oeq.TensorProduct takes 3 | MACE 0.3.15 conv_fusion call convention incompatible with oeq 0.6.6 |
| cuequivariance CUDA kernels | cublas >=12.5 required; torch needs 12.1 | Version conflict, no compatible build |
| cuequivariance naive fallback | 100-1000x slower (pathological einsums) | Unusable for production |
| TorchForce (compiled model) | Upstream abandoned for MACE in openmmml 1.6 | e3nn not TorchScript-compilable |

## Remaining Paths (Sub 1.3+ scope, diminishing returns)

1. **LAMMPS as MD engine** — MACE has native TorchScript/LAMMPS integration.
   Would eliminate PythonForce overhead (~5-10ms/step) entirely. Major refactor.
   Expected: ~15-25% improvement over current 2.56 ns/day.

2. **Newer MACE + oeq compatible version** — future MACE releases may fix the
   conv_tp calling convention for oeq. Would need version tracking.

3. **Rebuild OpenMM with CUDA 12** — for classical forces only; ~10-30% on <3ms
   classical component. Net impact: <5% overall.

## Impact on Schedule

At 2.56 ns/day, 5 ns per protein takes ~2.0 days. For 3 Tier B proteins
(WW, GB3, UBQ) running in parallel on separate H200 nodes, the wall-clock
is ~2.0 days from job start. This is well within the Sub 1.2 timeline (target
end 2026-05-16). **No schedule risk.** The bottleneck is now fair-share
recovery (account at 0.009, all jobs queued with Priority reason).
