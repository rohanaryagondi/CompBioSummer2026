# Cross-Agent Note: MACE NPT sbatch conda-path Regression Fix

## Summary

Task-001 v1 SLURM jobs 8885960 (WW, m4k2pz9q), 8885961 (GB3, n7r9tx4w),
8885962 (UBQ, q3j8vb6p) all FAILED in 4-5 s with exit code 1:0 at
2026-04-19T01:15-01:21. Root cause: `submit_mace_npt.sh` line 80
hardcoded `source /home/rag88/miniconda3/etc/profile.d/conda.sh` — a
path that does not exist on the cluster. The correct path for this
HPC is `/apps/software/system/software/miniconda/24.11.3/etc/profile.d/conda.sh`
(the Sub 1.1 working pattern, used by every `bioemu_*.sbatch` in Sub 1.1).

**Remediation (2026-04-19T01:37Z): 1-line sbatch fix.** Submit wrapper
edited; 6 failed logs renamed to `.failed`; 3 fresh SLURM jobs submitted
with new cryptic tags (8893817 t4x7qn2w, 8893818 f9b3rk6h, 8893819 p8m5dz1g)
to gpu_h200. env-mace itself was NOT modified (confirmed already
correctly used by Sub 1.1 working scripts). No Python / MACE code
changed.

---

## Root cause

### The broken path

`/home/rag88/miniconda3/` — `ls` reports "No such file or directory".
User has a `.conda/envs/` directory for env packages, but the
conda profile script lives inside the cluster's shared miniconda install,
not under the user's HOME.

### The working path (confirmed from 8 Sub 1.1 sbatch files)

`/apps/software/system/software/miniconda/24.11.3/etc/profile.d/conda.sh`

This is used verbatim by: `bioemu_batch_b200.sbatch`,
`bioemu_batch_h200.sbatch`, `bioemu_batch.sbatch`,
`bioemu_retry_cheap.sbatch`, `bioemu_retry_h200.sbatch`,
`bioemu_retry_long.sbatch`, `bioemu_retry_long_v2.sbatch`,
`bioemu_topup.sbatch` — all of which executed successfully in Sub 1.1.

### How the regression happened

The task-001 SubAgent hand-wrote the sbatch `--wrap` body rather than
copying verbatim from the Sub 1.1 reference `mace_hybrid_nvt.py` driver
script (which does not use an sbatch wrapper — it's invoked directly).
The MACE NPT submit path was a fresh `sbatch --wrap=` construction; the
conda path was a plausible-but-wrong guess.

### Why env-mace is fine

The env itself (`/home/rag88/.conda/envs/env-mace/`) is untouched and
correct. The `/home/rag88/.conda/envs/env-mace/bin/python` interpreter
works correctly when invoked with the right PYTHONPATH / PYTHONNOUSERSITE.
The failure was purely in the pre-Python shell activation step.

## Forward-looking recommendations

1. **All future SLURM scripts in this project MUST use
   `source /apps/software/system/software/miniconda/24.11.3/etc/profile.d/conda.sh`**
   for conda activation. Document in `shared/notes/operational-practices.md`
   alongside the PYTHONNOUSERSITE policy.

2. **Any new sbatch wrapper must be pattern-checked against a known-good
   Sub 1.1 script** before first submission. Even "obvious" shell
   idioms like conda activation can fail if the path differs from the
   site-specific cluster layout.

3. **Pre-submit smoke-test protocol (optional, but useful):** for any
   fresh SLURM wrapper, run `sbatch --test-only --wrap=...` to catch
   missing paths at submission time rather than at job start. Not done
   here because SLURM's --test-only may not catch runtime-only
   dereferences.

4. **No env-mace regression.** The SO3LR env-var fix note
   (`1.2-env-so3lr-typing-extensions-fix.md`) flagged concerns that
   MACE's `PYTHONNOUSERSITE=1` pattern could propagate incorrectly to
   SO3LR. That concern is orthogonal to this MACE failure — the MACE
   path itself is correct; only the conda shell script location was
   wrong.
