# Cross-Agent Note: GPU 0%-util root causes + efficiency paths

## Why we keep hitting "0% GPU util"

### MACE hybrid (Alpha-M track)

Every MD step in `mace_hybrid_npt.py` runs the cycle:

```
[GPU: MACE inference, ~20 ms via openmmml/openmmtorch TorchForce]
  →
[CPU: matscipy.neighbour_list rebuilds edges, ~20-100 ms, Python C extension]
  →
[GPU: OpenMM integrator step (PME + classical forces, ~1 ms)]
```

`matscipy.neighbour_list` is a compiled C function that does NOT release the
Python GIL during its execution. Concretely:

- Jobstats samples GPU utilization on ~1-second windows
- During the matscipy CPU phase (~50%+ of per-step wall), GPU is idle at 0%
- Jobstats can average a window where GPU was 0% the whole time → reported 0%
- Worse: our in-Python `_gpu_keepalive_thread` (5-min cadence matmul) is a
  Python daemon thread; it CANNOT execute while the main thread is in
  `matscipy.py_neighbour_list` because the GIL is held. So the in-script
  keepalive silently fails during long matscipy phases.

This is why 8939395/96/97 (MACE WW/GB3/UBQ v9) were YCRC-cancelled at 2-3 h
despite having an in-script keepalive.

### BioEmu (Gamma track)

First phase of every `bioemu_generate_single.py` run is a colabfold MSA build
(CPU-only, 5-30 min per new protein depending on sequence length). GPU is
legitimately idle during this phase. Denoising then runs GPU-bound at
~13-16 it/s per batch, so GPU util is high during generation phase.

Same GIL constraint as MACE — colabfold's C extensions hold the GIL during
MSA build, so Python threads can't run keepalive.

### SO3LR (Alpha-M track)

SO3LR vacuum NVT runs as a standalone `so3lr nvt` CLI subprocess. Our Python
wrapper just waits for subprocess to exit. The subprocess itself uses JAX,
which has its own neighbor-list implementation on GPU; SO3LR keeps GPU util
high continuously (~50-70%). **SO3LR does not hit the 0% GPU issue in our
setup**, which matches observations: all 5 SO3LR jobs ran to completion
without YCRC warnings.

## Fixes in place

### Short-term (deployed 2026-04-20)

1. **External-process GPU keepalive** in every submit wrapper:
   - `submit_mace_npt.sh` spawns `gpu_keepalive.py` as a bash background
     process BEFORE the main python call. It shares the SLURM GPU allocation
     but runs in a separate Python interpreter → not GIL-blocked.
   - `bioemu_batch2.sbatch` same pattern (patched 2026-04-20T02:15Z).
   - Keepalive does a 64x64 matmul every 300 s → registers >0% GPU util on
     YCRC's sampling, preventing auto-cancel.

2. **Sub 1.2 GPU watcher** (`sub12_gpu_watcher.sh`):
   - Persistent login-node bash script polling squeue every 5 min
   - Filters to Sub 1.2 cryptic-named jobs (8-char alphanumeric) to avoid
     touching other projects (rosetta `ros_*` etc.)
   - For any Sub 1.2 job not yet tracked, injects external-process keepalive
     via `srun --overlap` with the correct internal JobId (array tasks)
   - Safety net: even if a submit wrapper forgot to include the baked-in
     keepalive, the watcher catches it within 5 min

3. **MACE hang watchdog** (in `submit_mace_npt.sh`):
   - Background bash loop monitors the main stdout file's mtime
   - If no writes for 20 min, scancels the job so checkpoint/restart can
     recover (worst-case lost progress: ~3-7 min at 5K-step checkpoint
     interval vs previous 50 min at 100K-step interval)

### Longer-term (not yet deployed)

1. **Install `cuequivariance` / `cuequivariance-torch`** in env-mace.
   MACE logs warn: "cuequivariance or cuequivariance_torch is not available.
   Cuequivariance acceleration will be disabled." Installing this moves
   neighbor list + tensor-product computation onto the GPU itself. Expected
   impact:
   - Neighbor-list CPU phase eliminated → GPU utilization rises from ~50%
     to near-100%
   - Likely 2-3× speedup on per-step wall (matscipy is the bottleneck)
   - Removes the GIL-during-matscipy issue entirely — no more need for
     external keepalive on MACE jobs
   - Risk: env-mace was built non-destructively for D1 PASS; adding
     cuequivariance requires clone→v2 test first (operational-practices
     rule). Budget ~1-2 hr for install + smoke test.
   - Recommendation: schedule as a Sub 1.3 optimization pass if MACE
     production throughput becomes the critical-path bottleneck.

2. **Pre-cache BioEmu MSAs on CPU partition** (e.g., `day` or `scavenge`).
   MSA build is 100% CPU; running on a GPU partition wastes GPU SU (15/hr
   on RTX 5000 Ada while GPU idles for 5-30 min). Running on CPU-only
   partition: 0 SU for that phase.
   - Refactor: move `precache_msa.py` to a CPU-only sbatch. Once MSAs are
     cached, generation jobs start GPU-hot immediately.
   - Saves ~5-10 SU per protein × 93 proteins = 500-1000 SU per batch.
   - Also sidesteps the "YCRC auto-cancel for 0% GPU during MSA build"
     issue entirely (precache is legitimately CPU-only, so runs on CPU
     partition without YCRC GPU-util checks).
   - Recommendation: include in Sub 1.3 planning for any remaining batches.

3. **openmm-torch bridge** for MACE forces. Mentioned in Sub 1.1 Option 5
   follow-ups. Keeps force evaluation entirely in torch-CUDA without Python
   round-trips. Deferred in Sub 1.1 because MACE is inference-bound and the
   bridge wouldn't help much without also moving the neighbor list to GPU
   (i.e., needs cuequivariance too). Combined improvement from (openmm-torch
   + cuequivariance) could be 3-5× throughput.
