# Cross-Agent Note: env-delta Split into env-delta-v2 + env-cpa

## Finding

Sub 1.1 task-006 CPA install silently downgraded three load-bearing packages in
the shared `env-delta` conda env:

| Package | Pre-CPA (env-delta.yml) | Post-CPA (observed) | Delta impact |
|---|---|---|---|
| numpy | 2.2.6-ish | **1.23.5** | major (modern tools want >=1.26 or >=2.0) |
| anndata | 0.11.4 | **0.9.2** | major (API surface changed in 0.11) |
| scanpy | 1.11.5 | **1.10.2** | minor (still in Tahoe-x1's >=1.9,<2.0 range) |

Plus transitive downgrades: numba 0.65→0.58, llvmlite 0.47→0.41, umap-learn 0.5.12→0.5.7.

These downgrades are CPA's own hard pins (CPA 0.8.8 was released 2023 and has not
been updated). Sub 1.2 will install scFoundation and Tahoe-x1. Tahoe-x1 requires
`scanpy>=1.9,<2.0` (passes) and `torch>=2.5,<3.0` (passes). scFoundation has no
version pins but is Jupyter-notebook-heavy and typically used with modern AnnData.

To unblock Sub 1.2 without breaking the Sub 1.1 GEARS/scGPT/CPA baselines, I split
env-delta into two environments, **non-destructively** (the original env-delta is
untouched and remains the fall-back for CPA work):

1. **`env-delta-v2`** — modern stack for GEARS, scGPT, scFoundation, Tahoe-x1.
   - Python 3.10.20, numpy 1.26.4, scipy 1.14.1, pandas 2.2.2, anndata 0.11.4,
     scanpy 1.10.2, torch 2.11.0+cu128, networkx 3.3, cell-gears 0.1.2.
   - NOTE: GEARS install forces numpy 2.2.6→1.26.4, scanpy 1.11.5→1.10.2,
     scipy 1.15.3→1.14.1 (less severe than CPA's pins; anndata survives at 0.11.4).
2. **`env-cpa`** — isolated env for CPA only (not yet instantiated — yml ready).
   - anndata 0.9.2, scanpy 1.10.2, numpy 1.23.5, cpa-tools 0.8.8.

## Critical Gotcha During env-delta-v2 Build

Default `pip install torch` now returns `torch 2.11.0+cu130` wheels. McCleary
driver is 570.195 (CUDA 12.8). cu130 torch fails with `CUDA initialization: NVIDIA
driver is too old (found 12080)`. **Must pin the pip index** to get cu128 wheels:

```bash
pip install torch==2.11.0 --index-url https://download.pytorch.org/whl/cu128
```

This is now documented in `env-delta-clean.yml` build procedure.

## scFoundation + Tahoe-x1 Compatibility Scout

| Tool | Python | torch | anndata | scanpy | numpy | Hard pins? |
|---|---|---|---|---|---|---|
| env-delta-v2 | 3.10.20 | 2.11.0+cu128 | 0.11.4 | 1.10.2 | 1.26.4 | (target env) |
| env-cpa | 3.10 | 2.0.1+cu121 (TODO) | 0.9.2 | 1.10.2 | 1.23.5 | (target env) |
| scFoundation | any | no pin (needs cuda torch) | not pinned | not pinned | not pinned | NO (README lists bare names only) |
| Tahoe-x1 | >=3.10 | **>=2.5,<3.0** | (implicit via llm-foundry) | **>=1.9,<2.0** | (implicit) | YES (setup.py) |

**Conclusions:**

- **scFoundation** in env-delta-v2: LOW RISK. No pins to violate. Needs `einops` +
  `local_attention` (both pip-installable) — can add in Sub 1.2 Wave 1.
- **Tahoe-x1** in env-delta-v2: MEDIUM RISK. torch 2.11 SATISFIES >=2.5,<3.0.
  scanpy 1.10.2 SATISFIES >=1.9,<2.0. But Tahoe-x1 depends on
  `llm-foundry[gpu]>=0.17.1,<1.0` which pulls `flash-attn` + `composer` +
  `mosaicml-streaming`. flash-attn may only ship for torch 2.4–2.6 + cu124. If
  Tahoe-x1 install fails because of flash-attn wheel incompatibility, fall back
  to a separate `env-tahoex1` with torch 2.5.0+cu124. Their official guidance is
  to use Docker (`ghcr.io/tahoebio/tahoe-x1:latest`, based on
  `mosaicml/llm-foundry:2.2.1_cu121_flash2`) — this may not be usable on YCRC
  without Singularity/Apptainer wrapping.
- **CPA** in env-cpa: will be clean (isolated).

## Impact

  scripts work unchanged. If scFoundation install in Wave 1 triggers regressions,
  create env-scfoundation rather than contaminate env-delta-v2.
- **Combined paper:** no impact — this is pure infrastructure work.

## Recommended Action

1. **For Sub 1.2 HeadAI:** activate `env-delta-v2` for scFoundation install (Wave
   task) and the Tahoe-x1 install (separate task). If flash-attn blocks Tahoe-x1,
   escalate — possible env-tahoex1 creation with torch==2.5+cu124.
2. **For Sub 1.2 CPA re-use:** activate `env-cpa` (build from
   `phases/phase-0/subphase-0.1/envs/env-cpa.yml`) for any new CPA runs. The
   existing CPA adapter scripts in `phases/phase-1/subphase-1.1/output/scripts/`
   should be revalidated against env-cpa before Sub 1.2 CPA tasks.
3. **Do NOT touch env-delta.** It is a frozen rollback artifact. Future Delta
   tasks must use env-delta-v2 or env-cpa.
4. **Update task SLURM templates** in Sub 1.2 to use `conda activate env-delta-v2`
   (currently task-004 and task-006 scripts hardcode `env-delta`).

## Fallback / Rollback

If env-delta-v2 misbehaves at any later Sub 1.2 task:

- `conda remove -n env-delta-v2 --all` (safe — original env-delta is preserved)
- Rebuild with `conda env create -n env-delta-v2 -f env-delta-clean.yml` + the
  three pip install commands documented in env-delta-clean.yml
- For emergency fallback, Sub 1.2 can run GEARS in the original env-delta (it
  still works there).
