## Tahoe-x1 install

### Attempt 1 — env-delta-v2 — FAILED (as predicted)

- **Approach:** `pip install tahoe-x1 --no-build-isolation` in env-delta-v2 (torch 2.11.0+cu128).
- **Failure:** flash-attn 2.7.4.post1 (hard pin from llm-foundry 0.20+) has no prebuilt wheel for torch 2.11 / cu128. PyPI ships sdist only; compile requires nvcc. Available prebuilt wheels (from GitHub release v2.7.4.post1, 50 assets, verified via GitHub API) cover only torch 2.2-2.7 × cu12 (cu121 binary).
- **Conclusion:** env-delta-v2's torch 2.11.0+cu128 is outside the flash-attn compatibility matrix. Full log: `phases/phase-1/subphase-1.1/output/delta-tier1/tahoe_x1_attempt1_log.md`.

### Attempt 2 — env-tahoex1 (isolated) — PASS

- **Env created:** `env-tahoex1` from `phases/phase-0/subphase-0.1/envs/env-tahoex1.yml` (yml now documents the full build procedure).
- **Stack:** Python 3.10, torch 2.5.1+cu121, flash-attn 2.7.4.post1 (cu12torch2.5cxx11abiFALSE wheel — GLIBC_2.14, works on RHEL 8.10 / GLIBC 2.28), llm-foundry 0.17.1 (torch 2.5.1 pin), tahoe-x1 1.0.0, scanpy 1.11.5, anndata 0.11.4.
- **GLIBC gotcha (documented):** flash-attn 2.7.4.post1 cu12torch2.7 wheel requires GLIBC_2.32 (not on McCleary). The cu12torch2.6 wheel has the same issue. Only the cu12torch2.5 wheel stays on GLIBC_2.14, so torch must be pinned to 2.5.1. This forces llm-foundry to 0.17.1 (other versions pull torch 2.6/2.7).
- **llm-foundry flash-attn API patch:** flash-attn 2.7+ changed `unpad_input` from a 4-tuple to a 5-tuple return (added `seqused`). llm-foundry 0.17.1 (written for flash-attn 2.6.x) does a hard 4-tuple unpack. Patched `llmfoundry/models/mpt/modeling_mpt.py` lines 292-303 to do a variadic unpack and ignore the 5th element. This patch is documented in the yml header and at the top of the patched function.
- **ComposerTX.from_hf vs get_batch_embeddings API nuance:** `ComposerTX.from_hf()` returns the `ComposerTX` wrapper whose `.forward()` takes a `batch` dict. `tahoe_x1.tasks.emb_extractor.get_batch_embeddings()` internally calls `model(genes=..., values=..., ...)` with kwargs matching the *inner* `TXModel.forward`. Pass `model.model` (the TXModel attribute of ComposerTX) to `get_batch_embeddings`.
- **70m model smoke test (SLURM 8708255):**
  - GPU: NVIDIA RTX 5000 Ada (31.5 GB)
  - Load from HuggingFace (`tahoebio/Tahoe-x1`, `model_size="70m"`): 1.1 s
  - Params: 70,996,993. Vocab size 62,720 (includes `<cls>`, `<pad>`, etc. special tokens on top of the 62,710 Tahoe-100M gene IDs).
  - Tahoe-100M vocab mapping: **100.0% (62,710/62,710)** — Tahoe-x1 was trained on Tahoe-100M so every ensembl_id resolves.
  - Forward pass (100 cells, max_length=2048): 0.9 s, output shape `(100, 512)` float32 (cell_emb dim = 512 for 70m).
  - Peak GPU memory: **0.72 GB** — trivial on RTX 5000 Ada.
- **3B model smoke test (SLURM 8709432, RTX 5000 Ada — H200 cancelled after profiling 3B footprint):**
  - GPU: NVIDIA RTX 5000 Ada (31.5 GB) — chosen after seeing 70 m used 0.7 GB, extrapolating that 3B inference at bf16 ≈ 10-15 GB.
  - Load from HuggingFace: 85.6 s (10.9 GB safetensors download + state_dict load).
  - Params: **2,704,463,361** (≈2.7 B — the README's "3b" label is approximate).
  - Vocab size 62,721; Tahoe-100M mapping **100% (62,710/62,710)**.
  - Peak GPU memory: **16.7 GB** (alloc 10.1 GB model weights + 6.6 GB activations). Fits RTX 5000 Ada (32 GB) with 14+ GB headroom.
  - Forward pass (100 cells, max_length=2048, batch_size=8, num_workers=2): **8.5 s**.
  - Cell embeddings shape `(100, 2560)` float32 (cell_emb dim = 2560 for 3b).
  - Wall time 2:02 (mostly HF download). SU cost ~0.5 (RTX 5000 Ada, 15 SU/hr).
  - **Production recommendation for Sub 1.2:** Run Tahoe-x1 3B on RTX 5000 Ada (cheaper and faster to schedule than H200). For full Tahoe-100M inference (100 M cells) route to H200 only if batched RTX 5000 Ada throughput becomes the bottleneck; per-cell compute is ~85 ms at bs=8 on RTX 5000 Ada which gives ~14 days for 100 M cells. For a 5 M-cell subset pilot (Delta's typical scope) this is ~17 GPU-hours — tractable on RTX 5000 Ada (~255 SU).
