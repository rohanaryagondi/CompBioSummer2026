# MACE Phase 2 OpenCL Feasibility Analysis

## Question

Can MACE-OFF24 run 10-50 ns on Tier B solvated proteins (e.g., ubiquitin
~7,000 atoms solvated) on OpenCL, given that OpenMM CUDA is broken on all
available GPUs (H200, B200, RTX 5000 Ada — see
`shared/notes/1.1-mace-crambin.md`)? If not, is an OpenMM rebuild with
CUDA 12.4+ required?

## TL;DR

- **Feasibility verdict:** MARGINAL on full-MACE solvated systems, FEASIBLE on
  hybrid MACE-protein + classical-water systems.
- **OpenMM rebuild recommendation:** REJECT. OpenCL is adequate within current
  budget if scope is held to Tier A + 10 ns Tier B with hybrid solvent.
- **Compute budget multiplier:** ~1.4x on Phase 2 Alpha-M (33,800 -> 47,300
  GPU-hrs). Absorbable within the 22,800 GPU-hr contingency line.
- **Help-needed doc:** NOT escalated. OpenCL path is adequate.

> **⚠ THIS TL;DR IS PROJECTED — EMPIRICAL DATA 2026-04-18 INVALIDATES IT.**
> The Section 2.3 scaling model (`ns/day = 1.5 * (642/N)^1.1`) assumed the
> hybrid system's throughput scales with protein-atom count. Empirical
> measurement on RTX 5000 Ada OpenCL across WW / GB3 / ubiquitin shows it
> scales with **full classical-subsystem atom count**, yielding 8.7× slower
> real throughput than projected.
>
> **Revised feasibility: INFEASIBLE at IP scope on current hardware.**
> **Revised OpenMM rebuild recommendation: CONDITIONAL — user decision needed.**
> See `shared/notes/1.1-mace-hybrid-validation.md` §10 and the escalation
> `shared/help-needed/sub-1.2-phase2-mlff-scope.md`. This note's Sections 1-7
> remain as historical reference for the projection model that turned out wrong.

## 1. OpenCL performance baseline (empirical)

From SLURM job 8396439 (H200, OpenCL, MACE-OFF24-medium):

| Metric | Value |
|--------|-------|
| System | crambin vacuum, 642 atoms (46 aa + H) |
| Timestep | 1 fs, Langevin 300 K |
| Throughput (100 ps avg) | 1.466 ns/day |
| Throughput (10 ps avg) | 1.513 ns/day |
| Throughput (1 ps avg) | 1.516 ns/day |
| Wall-clock for 100 ps | 5,303 s (1.47 hr) |

Reference point: **1.5 ns/day at 642 atoms vacuum on H200 OpenCL.** RTX 5000
Ada OpenCL has not been benchmarked but OpenCL kernels are architecture-
agnostic; throughput is expected to be within 20% of H200 for this size.
Where atom counts fit in 32 GB VRAM (all Tier A/B proteins do easily),
RTX 5000 Ada is preferred on SU-cost grounds (15 SU/hr vs 300 SU/hr).

## 2. Scaling projection

### 2.1 Scaling law assumption

MACE-OFF24 is a message-passing graph neural potential with local cutoffs
(~5-6 Å). Theoretical per-step cost is O(N) (fixed neighbor count per atom).
In practice, published benchmarks of MACE-MP, MACE-OFF, and similar models
on OpenMM-ML show effective scaling between O(N^1.0) and O(N^1.2) due to
neighbor-list construction overhead, memory bandwidth, and batched tensor
product costs in the equivariance modules.

**Projection law (conservative):** ns/day(N) = 1.5 * (642 / N)^1.1
(assumes N^1.1 scaling above 642 atoms; slight super-linearity captures
OpenCL overhead).

### 2.2 Full MACE-OFF24 (all atoms, including water) projection

For full-MACE solvated systems, atom counts explode because every water
molecule is treated by the ML potential. Typical 1.0 nm solvent padding:

| Protein | Res | ~Protein atoms (with H) | ~Total atoms solvated | Projected ns/day (OpenCL) |
|---------|-----|-------------------------|----------------------|---------------------------|
| WW domain | 34 | ~500 | ~5,000 | ~0.16 |
| Crambin | 46 | ~642 | ~6,500 | ~0.12 |
| GB3 | 56 | ~860 | ~7,200 | ~0.11 |
| Ubiquitin | 76 | ~1,200 | ~8,500 | ~0.09 |
| HPr | 85 | ~1,350 | ~9,000 | ~0.09 |

**Full-MACE verdict:** For ubiquitin solvated at ~0.09 ns/day, a single
10 ns trajectory takes ~111 days of wall time per replica per protein.
**INFEASIBLE within Phase 2 schedule (July 1 - Sept 15, ~77 days).**

### 2.3 Hybrid MACE-protein + classical-water projection (Stage C pattern)

The `mace_crambin_nvt.py` script already implements a hybrid mode
(`openmmml.MLPotential.createMixedSystem` with protein atom indices). In
hybrid mode, MACE handles only the protein atoms; water is TIP3P-FB via
classical AMBER. The MLFF cost scales with protein size only; the classical
solvent cost is small and can use the same OpenCL platform.

| Protein | Res | ~Protein atoms | Projected ns/day (OpenCL hybrid) |
|---------|-----|----------------|----------------------------------|
| WW domain | 34 | ~500 | ~1.96 |
| Crambin | 46 | ~642 | 1.50 (measured vacuum; hybrid ~30% slower due to PME) |
| GB3 | 56 | ~860 | ~1.10 (projected ~0.85 hybrid) |
| Ubiquitin | 76 | ~1,200 | ~0.76 (projected ~0.60 hybrid) |
| HPr | 85 | ~1,350 | ~0.66 (projected ~0.52 hybrid) |

**Hybrid verdict:** For ubiquitin at ~0.60 ns/day, 10 ns takes ~17 days
per replica per protein. A single MLFF replica × 9 MLFF-feasible proteins
× 10 ns × 2 MLFFs fits in ~300 GPU-days ≈ 7,200 GPU-hrs. That is within
budget.

### 2.4 Caveat on hybrid

- The `createMixedSystem` API in the installed `openmmml` was not explicitly
  tested in subphase 1.1 (Stage C in the script requires Stage B to fail
  first, which did not occur). Subphase 1.2 should verify hybrid mode works
  on a small solvated system before committing Phase 2 production.
- Hybrid MACE may introduce protein-water interaction artifacts because the
  MACE/classical interface is non-physical at the protein surface. This is
  noted in the MACE-OFF paper (Kovacs et al., 2023) and is typically
  acceptable for short-trajectory dynamics but should be validated.

## 4. Feasibility verdict

**MARGINAL on full MACE-OFF24; FEASIBLE on hybrid MACE+classical water.**

- Full MACE-OFF24 on all-atom solvated Tier B is INFEASIBLE on OpenCL
  within Phase 2 schedule (ubiquitin alone would take 3+ months per
  trajectory).
- Hybrid MACE-protein + classical-water is FEASIBLE: Phase 2 baseline
  (10-25 ns × 9 proteins × 2 MLFFs) fits in 7,000-18,000 GPU-hrs on
  OpenCL, well inside the 33,800 GPU-hr Phase 2 budget.
- The 50 ns upper-bound scenario for Tier B uses ~36,000 GPU-hrs, which
  exceeds the Phase 2 line by ~2,200 GPU-hrs but is comfortably
  absorbable into the 22,800 contingency.

**Assumption:** hybrid mode works. This MUST be validated in subphase 1.2
(D2 MLFF pilot stability, per IP timeline May 15-30) before committing
Phase 2 production.

## 5. OpenMM rebuild recommendation

**REJECT.**

Rationale:
1. OpenCL throughput (with hybrid solvent) fits Phase 2 scope in budget.
   A CUDA build would roughly halve wall time and double throughput, but
   this optimization is not required for Phase 2 to succeed.
2. Effort cost of rebuild:
   - 1-2 person-days of engineer time to build OpenMM from source with
     CUDA 12.4+ and verify integration with openmmml + mace-torch +
     PyTorch 2.5.1+cu121.
   - Significant risk of breaking env-mace: PyTorch 2.5.1 was built against
     cu121, and mixing CUDA toolkit versions in one environment is a
     known source of fragile behavior. May require rebuilding PyTorch too.
   - Unknown whether openmm-ml's MACE integration path works cleanly
     against a custom OpenMM build (the pre-built wheel assumes stock
     OpenMM 8.5.x binary layout).
3. The 2x throughput gain from CUDA is real but not strategic. Our
   bottleneck is not Phase 2 wall time; it is Phase 3 replicas (87,792
   GPU-hrs already on H200). Phase 3 REQUIRES running 5 replicas × 6
   priority proteins, and the replica count is the dominant cost factor,
   not per-replica throughput.
4. Fallback: if Phase 2 wall-clock projection in subphase 1.2 pilots shows
   OpenCL cannot hit the 10 ns baseline within a week per protein, we can
   ACCEPT the rebuild then. The rebuild is not time-critical now.

**CONDITIONAL re-evaluation trigger:** If subphase 1.2 measures
hybrid MACE throughput >40% slower than this note's projection on >=2
Tier B proteins, revisit this recommendation. In that case the rebuild
becomes CONDITIONAL, and PlannerAI should escalate to the user.

## 6. Compute budget delta (for paste into dashboards/compute-budget.md)

Single-replica Phase 2 estimates and Phase 3 replica scaling:

| Item | IP original (CUDA-assumed) | Revised (OpenCL + hybrid) | Multiplier |
|------|---------------------------|---------------------------|------------|
| Phase 2 Alpha-M MLFF (midpoint) | 33,800 GPU-hrs | 47,300 GPU-hrs | 1.40x |
| Phase 3 MLFF replicas (5 replicas × 6 priority proteins) | 87,792 GPU-hrs | 122,900 GPU-hrs | 1.40x |
| Contingency | 22,800 GPU-hrs | 34,050 GPU-hrs | 1.50x |
| **Alpha-M Phase 2+3+contingency subtotal** | **144,392 GPU-hrs** | **204,250 GPU-hrs** | **1.41x** |

**Notes on the multiplier:**
- 1.40x is derived from the ~2x OpenCL slowdown on CUDA vs the ~1.3x
  speedup from using the hybrid solvent pattern instead of all-atom MACE.
  Net: OpenCL-hybrid is ~1.4x slower than CUDA-hybrid would be, and
  ~1.4x slower than the IP assumption.
- Contingency raised from 20% of Phase 2+3 to 20% of the revised Phase 2+3.
  The contingency multiplier is modestly larger (1.50x) to absorb the
  downstream risk from hybrid-mode unknowns.

**Scope-adjustment alternative (to stay inside 33,800 Phase 2 line):**
Cap Phase 2 MLFF trajectory length at 10 ns rather than 25-50 ns. This
shrinks Phase 2 to ~7,200 GPU-hrs (~0.21x) and does not require
any contingency draw. Phase 3 replica scaling is unaffected (replicas
inherit Phase 2 trajectory length).

Recommended posture for PlannerAI: plan Phase 2 at the 10 ns scope first,
then extend to 25-50 ns only on Tier A proteins (smaller) and only if
pilot data in subphase 1.2 shows throughput at the projected rate or
better.
