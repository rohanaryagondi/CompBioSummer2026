# Cross-Agent Note: env-so3lr SLURM Env-Var Regression Fix (typing_extensions)

after CLI returned exit 1). Root cause: `so3lr_pilot.sbatch` set

## Root cause analysis

### The `PYTHONNOUSERSITE=1` provenance

The v1 `so3lr_pilot.sbatch` header comment said:
> `PYTHONNOUSERSITE=1 (Sub 1.1 lesson: numpy MUST be from conda env)`

This comment attribution is **incorrect**. Sub 1.1's lesson about
`PYTHONNOUSERSITE=1` applies to **env-mace** (MACE's numpy 2.x is
ABI-incompatible with user-site numpy; MACE SLURM scripts set the flag
to force conda numpy). Grepping Sub 1.1 scripts confirms env-mace
scripts (e.g., `mace_hybrid_ww.sbatch`, `mace_implicit_gb3.sbatch`,
`mace_crambin.sbatch`) set the flag, while **env-so3lr scripts
explicitly DO NOT**:

- `so3lr_crambin_v5.sbatch` (the 2026-04-16 D1-PASS path): no flag.
- `so3lr_crambin_cli.sbatch`: `export PYTHONNOUSERSITE=0` (explicitly).
- `so3lr_quick_test.sbatch`: `export PYTHONNOUSERSITE=0` (explicitly).

All three prepend `/home/rag88/.local/lib/python3.12/site-packages` to
`PYTHONPATH`. That is the Sub 1.1 working SO3LR contract.

### Env dependency survey

When `PYTHONNOUSERSITE=1` is set and user-site is hidden,
`from so3lr import So3lrCalculator` fails iteratively at:
1. `typing_extensions` (flax/struct.py:23)
2. `rich` (flax/linen/summary.py:30)
3. `yaml` (flax/linen/summary.py:33)
4. `pandas` (mlff/utils/evaluation_utils.py:7)
5. `dateutil` (pandas/__init__.py, transitive)

None of these are in `/home/rag88/.conda/envs/env-so3lr/lib/
python3.12/site-packages/`. All 5 are present in user-site at standard
versions (typing_extensions 4.15.0, rich 14.3.3, pandas latest,
pyyaml 6.x, python-dateutil 2.x). env-so3lr depends on user-site for
its Python stdlib-adjacent layer; numpy/jax/so3lr/mlff itself ARE in
env-so3lr and take precedence via env-first PYTHONPATH.

## Fix method: Option C (sbatch env-var change)

### Options considered

| Option | Cost | Risk | Chosen |
|-------|------|------|--------|
| **A. Clone env-so3lr â†’ v2 + pip install missing deps** | 5.8 GB copy, 15 min install | Version skew vs Sub 1.1 crambin baseline; new env name must be threaded through scripts | no |
| **B. pip install missing deps in place into env-so3lr** | 30 s, 4-5 packages (typing_extensions, rich, pyyaml, pandas, dateutil) + their transitive deps | Non-trivial version shifts vs user-site; Sub 1.1 crambin tested against user-site versions, not new env versions. Violates "additive pure-Python only" rule (pandas/dateutil are not pure-Python). | no |
| **C. Fix sbatch env-vars (unset PYTHONNOUSERSITE, add user-site to PYTHONPATH AFTER env site-packages)** | 2 edited lines, 0 package installs | Matches Sub 1.1 working pattern exactly. env untouched. | **YES** |

Option C is non-destructive in the strongest sense: nothing in
`/home/rag88/.conda/envs/env-so3lr/` changed. Only
`output/scripts/so3lr_pilot.sbatch` and
`output/scripts/submit_so3lr_pilot.sh` changed (+ this note and the
stability note + the status report).
