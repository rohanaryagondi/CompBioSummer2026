## Root-cause diagnosis (confirmed)

### Why the env-mace CUDA platform fails

env-mace's OpenMM CUDA plugin (`lib/plugins/libOpenMMCUDA.so`) was installed from conda-forge's `openmm-8.5.1-py310h39b4bc9_0` build variant which links against `libnvrtc.so.13` (CUDA 13.2). The bundled libnvrtc emits PTX ISA 8.6+ (CUDA 13 default). Driver 570.195.03 supports up to PTX ISA 8.4 (CUDA 12.8). Result: every OpenMM CUDA kernel compile emits `CUDA_ERROR_UNSUPPORTED_PTX_VERSION (222)`.

This is an ISA version check at the driver level, not a compute-capability (sm_XX) check. All three GPU families fail for the same reason.

### Confirmation via rebuild

When OpenMM 8.5.1 is rebuilt against CUDA 12.8 module (which exposes `libnvrtc.so.12.8.61`), its CUDA plugin links against `libnvrtc.so.12` SONAME. Loading OpenMM's CUDA context on the same RTX 5000 Ada node that previously failed now succeeds:


(Compare to the FAILED probe on same machine with env-mace's libnvrtc-13 plugin: `Platform CUDA FAILED: OpenMMException: Error loading CUDA module: CUDA_ERROR_UNSUPPORTED_PTX_VERSION (222)`.)

**The OpenMM CUDA platform itself works.** The failure is downstream: during minimization, openmm-ml 1.6's `PythonForce` invokes a MACE forward pass via TorchScript, and that MACE call runs a CUDA kernel on PyTorch's CUDA context. OpenMM has its own separate CUDA context. When OpenMM queries the simulation state after MACE has run, a CUDA stream or buffer handle is invalid — this is a classic multi-context interop issue.

### Why the openmm-ml 1.6 PythonForce path is fragile

openmm-ml 1.6 moved away from `openmm-torch`'s C++ TorchForce (which shared CUDA context with OpenMM) to a pure-Python `PythonForce` that invokes TorchScript-compiled MACE models. This avoided the need for a C++ `libopenmm-torch.so` build but introduced two-context CUDA juggling. Typical fixes include:

1. `CUDA_VISIBLE_DEVICES=0` both in OpenMM and PyTorch (ensure same device ordinal).
2. `torch.cuda.synchronize()` before each OpenMM state query.
3. Configure OpenMM's CUDA Platform to use the same CUDA stream as PyTorch (`DeviceIndex` property).
4. Build openmm-torch 1.5.1 from source against the same pip PyTorch 2.5.1+cu121 (gives a GPU-aware C++ TorchForce that shares context).

Options 1-3 are drop-in; Option 4 is the "clean" fix but adds another 1-2 hr of build work.
