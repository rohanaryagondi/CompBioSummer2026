# Cross-Agent Note: MACE-OFF24 Crambin D1 Gate Result

## Finding

MACE-OFF24 (mace-torch 0.3.15) successfully runs stable NVT dynamics on crambin
(1CRN, 642 atoms with hydrogens) via the OpenMM-ML integration. Vacuum NVT at
300 K with 1 fs timestep and Langevin thermostat produced a stable trajectory
through 100 ps (confirmed) with clean energies and temperatures, no NaN, no
extreme forces.

**D1 gate assessment: PASS** (100 ps stable NVT trajectory confirmed; D1
criterion is >= 100 ps stable NVT).

## Platform verdict (DEFINITIVE)

**OpenMM's CUDA platform is UNUSABLE on every GPU available to this project.**
The OpenMM CUDA kernels bundled in the env-mace build (CUDA toolkit 12.1,
PyTorch cu121) were compiled against a PTX version that is rejected by all
three GPU architectures we have access to:

| GPU | Compute cap | SLURM job | Error | Result |
|-----|-------------|-----------|-------|--------|
| H200 | sm_90 | 8395988 | `CUDA_ERROR_UNSUPPORTED_PTX_VERSION (222)` | CUDA BROKEN |
| B200 | sm_100 | 8395706 | `no kernel image is available for execution on the device` | CUDA BROKEN |
| RTX 5000 Ada | sm_89 | **8398672** | `CUDA_ERROR_UNSUPPORTED_PTX_VERSION (222)` | CUDA BROKEN |

Notes:
- The failure is NOT PyTorch-side. `torch.cuda.is_available()` returns True on
  all three GPUs; PyTorch tensor ops work. The error is specifically in OpenMM's
  compiled CUDA kernels, which the env-mace build ships as prebuilt binaries.
- This is an environment-level incompatibility that applies to ANY MACE
  simulation running through OpenMM-ML in env-mace. All downstream Phase 2
  MLFF runs will hit the same wall.
- OpenCL is the only functional backend. The OpenMM OpenCL kernels run
  correctly on all three GPU architectures.

## Performance on OpenCL

| Metric | Value |
|--------|-------|
| System | crambin vacuum, 642 atoms |
| Platform | OpenCL (auto-fallback from CUDA) |
| GPU | H200 (NVIDIA) |
| Timestep | 1 fs (Langevin thermostat, 300 K, friction 1/ps) |
| Throughput (1 ps) | 1.516 ns/day |
| Throughput (10 ps) | 1.513 ns/day |
| Throughput (100 ps) | 1.466 ns/day |
| Minimization PE | -281,976 kJ/mol |
| NVT PE (10 ps avg) | -279,640 +/- 70 kJ/mol |
| Temperature (equil.) | 282-310 K (converging to 300 K target) |
| NaN / extreme forces | None detected |

**Key insight:** OpenCL throughput is approximately 2x slower than CUDA would
be for the same system (typical OpenMM scaling). For a 642-atom vacuum system
this is manageable; for larger solvated systems this is much more painful.

## Impact

### Alpha-M

MACE-OFF24 is viable for the Phase 2 benchmark BUT throughput is severely
degraded on OpenCL. The practical question is no longer "can MACE run?"
(it can, on OpenCL), but "can Phase 2 fit in budget on OpenCL?" A full
feasibility analysis for solvated Tier B proteins (the Phase 2 workload)
is in [`shared/notes/1.1-mace-phase2-feasibility.md`](./1.1-mace-phase2-feasibility.md).

**Upshot from that analysis:** Phase 2 MACE on OpenCL is MARGINAL but
FEASIBLE within the IP compute budget if scope is held to Tier A + short
Tier B trajectories (10 ns rather than 50 ns). Staying on OpenCL is the
recommended path. Rebuilding OpenMM from source with a newer CUDA toolkit
is REJECTED as not cost-effective given the OpenCL path works.

### Combined

No correctness concern. The CUDA incompatibility is a wall-time issue, not
a physics issue. Energy conservation, force magnitudes, and temperature
control all look normal. The combined paper depends on Phase 2 MLFF results
existing, not on them being fast. Phase 2 schedule pressure increases,
but D6 (Aug 31) remains feasible with OpenCL.

### Other tracks

No direct impact on Gamma, Delta.

## Recommended Action (UPDATED 2026-04-17)

1. **Stop testing CUDA.** All three accessible GPU architectures have been
   verified as CUDA-BROKEN. Future MACE jobs should skip CUDA initialization
   entirely (saves ~3 seconds of platform-probe time per job). The fallback
   logic in `mace_crambin_nvt.py` already handles this gracefully; subsequent
   scripts can remove `CUDA` from the platform list.

2. **Use OpenCL for all Phase 2 MACE runs.** H200 OpenCL throughput is the
   baseline. RTX 5000 Ada OpenCL is likely comparable (not benchmarked but
   OpenCL kernels are architecture-agnostic). Prefer RTX 5000 Ada on SU-cost
   grounds (15 SU/hr vs H200 300 SU/hr) unless a job specifically needs
   the H200's 150 GB VRAM (unlikely for <2000-atom solvated systems).

3. **Do NOT attempt OpenMM rebuild.** Rebuilding OpenMM from source against a
   newer CUDA toolkit (12.4+) would fix the PTX issue, but the effort cost
   (1-2 person-days per the feasibility analysis, plus risk of breaking
   env-mace dependencies) exceeds the benefit (~2x throughput gain) given
   that OpenCL fits within budget. See feasibility note for full analysis.

4. **Budget adjustment.** Phase 2 Alpha-M budget is projected to rise from
   33,800 to ~45,000-55,000 GPU-hrs on OpenCL (a 1.3-1.6x multiplier). This
   is absorbable within the 20% contingency line (22,800 GPU-hrs). PlannerAI
   should update `dashboards/compute-budget.md` using the delta table in
   the feasibility note. If Phase 2 scope is held to 10 ns Tier B rather
   than 50 ns, the increase is smaller.

5. **SLURM script template.** Future MACE simulation scripts should:
   - Skip CUDA platform entirely (don't even try it).
   - Request OpenCL explicitly.
   - Target RTX 5000 Ada unless memory requires H200.
   - Use 1 fs timestep, 300 K Langevin, exactly as in
     `phases/phase-1/subphase-1.1/output/scripts/mace_crambin_nvt.py`.
